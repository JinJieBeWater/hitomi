# 考勤系统业务流程说明

## 概述

本文档描述当前毕设 MVP 考勤系统的完整业务流程。

系统基于当前仓库技术栈实现：

- 前端后台：Nuxt
- 后端接口：oRPC + Nitro API
- 数据库：Drizzle + SQLite
- 管理员登录：Better Auth
- 硬件设备：[`SZPI ESP32-S3` 开发板](https://wiki.lckfb.com/zh-hans/szpi-esp32s3/)（基于 `ESP32-S3`）

系统架构前提：

- 单组织
- 单服务
- 单数据库
- 只考虑组织内部的考勤场景

当前 MVP 补充约束：

- 当前只考虑单管理员使用场景
- 当前答辩演示按单设备部署实现
- 默认只使用中国时间，不处理其他时区和夏令时

系统目标是完成一个最小可用闭环：

1. 管理员登录后台
2. 管理员工
3. 注册考勤设备并分配密钥
4. 配置全局考勤时间段
5. 给设备下发录脸任务
6. 设备完成人脸录入
7. 设备识别员工并上传考勤记录
8. 后台查看考勤结果

## 系统角色

### 1. 管理员

管理员通过后台网页登录系统，负责：

- 维护员工信息
- 注册和管理设备
- 配置考勤时间段
- 发起员工录脸任务
- 查看考勤记录

### 2. 设备

ESP32 设备负责：

- 通过 `deviceCode + apiKey` 向服务端鉴权
- 同步当前配置
- 同步并本地保存员工信息
- 接收待处理录脸任务
- 采集和录入员工人脸
- 识别员工
- 本地判断是上班打卡还是下班打卡
- 本地缓存待上传考勤记录
- 本地记录接口拒绝原因和失败信息
- 在网络可用时上传考勤记录

### 3. 员工

员工不直接操作系统，只在设备前完成人脸录入和日常打卡。

## 核心业务流程

### 1. 管理员登录

管理员通过 Better Auth 登录后台。

登录成功后可以访问受保护页面，包括：

- 员工管理页
- 设备管理页
- 录脸记录页
- 考勤记录页

## 2. 员工管理流程

管理员在后台手动维护员工信息。

员工字段包括：

- 员工编号 `code`
- 员工姓名 `name`

业务规则：

- 员工编号必须唯一

### 3. 设备注册流程

管理员在后台创建设备。

系统在创建设备时自动生成：

- `deviceCode`
- `apiKey`
- `bootstrapSerial`
- `bootstrapSecret`

其中：

- `deviceCode` 用于标识设备
- `apiKey` 用于设备调用接口
- `bootstrapSerial` / `bootstrapSecret` 用于设备首配和待激活身份
- 数据库保存 `apiKeyHash`
- MVP 激活阶段允许服务端暂存“待设备领取”的运行时密钥，领取后立即清除
- 后台在创建设备当次展示明文 `apiKey` 与 bootstrap 凭据

业务规则：

- 每台设备有唯一 `deviceCode`
- 设备可被禁用
- 禁用后设备不可继续调用同步、录脸、考勤上传接口
- `apiKey` 在设备创建时生成，并在设备成功领取前保持为待领取态
- MVP 阶段不支持任意后台重置或轮换 `apiKey`

### 4. 考勤时间配置流程

系统只有一套全局考勤配置。

管理员在后台配置两个时间段：

- 上班打卡时间段
- 下班打卡时间段

示例：

- 上班：`09:00 - 10:00`
- 下班：`18:00 - 19:00`

约束：

- 后端不写默认时间
- 前端表单可提供默认显示值
- 首次保存时才在数据库中创建配置
- 时间统一按 `Asia/Shanghai` 解释
- 两个时间段不得重叠、不得相互包含、不得跨天
- 设备端与服务端统一按 `[start, end)` 方式判定时间段

### 5. 设备同步流程

设备启动后或定时向服务端发起同步请求。

请求携带：

- `deviceCode`
- `apiKey`

服务端校验设备身份成功后返回：

- 当前设备状态
- 全局考勤配置
- 设备本地识别所需的员工信息
- 当前设备待处理的录脸任务

设备通过同步接口得到：

- 最新考勤时间段
- 最新员工信息
- 是否允许继续工作
- 当前是否有待录入员工

补充说明：

- MVP 中设备启动时可全量拉取配置、员工信息和录脸任务
- 为简化实现，设备后续定时同步也可以继续使用全量返回

### 6. 录脸任务流程

管理员在后台为员工发起录脸任务。

当前设计约束：

- 一个员工只保留一条当前录脸记录
- 同一台设备同一时刻可存在多条 `pending` 的录脸任务

流程如下：

1. 管理员选择员工和设备
2. 服务端创建或更新该员工的 `face_profile`，状态设为 `pending`
3. 设备下次同步时拿到待录脸任务列表
4. 设备根据本地 UI / 操作流程选择其中一条任务进行录脸
5. 录脸完成后，设备将对应任务结果上报服务端
6. 服务端把对应记录状态改为 `success` 或 `failed`

说明：

- 服务端不保存人脸图片
- 服务端不保存人脸特征向量
- 人脸模板仅保存在设备本地
- 人脸模板库主存储固定为 `SD Card`
- 当前系统不依赖设备侧槽位号做身份识别
- 设备在后续识别成功时直接上传 `employeeId`
- 后台可取消 `pending` 状态的录脸任务，取消后状态为 `cancelled`
- 设备若在任务已取消后再上报录脸结果，服务端返回固定拒绝原因 `TASK_CANCELLED`，并忽略该次结果

### 7. 日常打卡流程

设备在本地完成人脸识别后，得到员工身份。

补充说明：

- 设备侧识别依赖本地模板库
- 模板库主存储位于 `SD Card`

识别成功后流程如下：

1. 设备得到 `employeeId`
2. 设备获取当前本地时间
3. 设备根据已同步的全局考勤配置判断：
   - 是否落在上班打卡时间段
   - 是否落在下班打卡时间段
4. 如果命中上班时间段，则判定为 `clock_in`
5. 如果命中下班时间段，则判定为 `clock_out`
6. 如果不在任何有效时间段内，则不作为有效打卡上传

### 8. 设备本地去重流程

设备需要考虑网络中断场景，因此可先本地缓存待上传记录。

本地去重规则：

- 同一天
- 同一个员工
- 同一种类型
- 只保留最早的一次

例如：

- 同一员工在 `2026-03-26` 上午时间段被识别多次
- 设备只保留最早的那次 `clock_in`

这样可以减少重复上传，也符合系统最终规则。

### 9. 考勤上传流程

当网络可用时，设备将缓存中的有效记录上传给服务端。

上传数据最少包含：

- `employeeId`
- `deviceId` 或通过设备鉴权识别
- `recognizedAt`
- `type`

服务端收到后执行以下校验：

1. 设备鉴权是否通过
2. 设备状态是否为 `active`
3. 员工是否存在
4. 当前 MVP 不额外校验员工与设备的绑定关系
5. `recognizedAt` 对应的本地时间是否落在对应时间段
6. 同一员工在同一天是否已经存在同类型记录

保存规则：

- 系统只保存有效考勤记录
- 同一员工同一天同一类型只保留一条
- 如果后上传的记录时间更早，则更新原记录为更早时间
- 服务端拒绝写入时需要返回稳定的拒绝原因，设备负责本地展示或记录

### 10. 后台查看考勤记录流程

管理员在后台查看考勤结果。

支持按以下维度筛选：

- 日期
- 员工
- 类型：上班或下班
- 设备

页面展示信息包括：

- 员工编号
- 员工姓名
- 打卡类型
- 打卡时间
- 设备名称

## 页面规划

### 1. Dashboard

展示基础统计信息，例如：

- 员工总数
- 设备总数
- 今日上班打卡数
- 今日下班打卡数

并直接包含全局考勤配置区块，支持：

- 查看当前配置
- 编辑上班时间段
- 编辑下班时间段
- 首次创建全局配置

### 2. 员工管理页

支持：

- 创建员工
- 编辑员工
- 查看员工当前录脸状态

### 3. 设备管理页

支持：

- 创建设备
- 查看设备码
- 查看初始化密钥
- 修改设备名称
- 启用或禁用设备
- 查看最近在线时间

### 4. 录脸记录页

支持：

- 为员工选择设备并创建录脸任务
- 查看录脸状态
- 重新发起失败任务

### 5. 考勤记录页

支持：

- 按日期筛选
- 按员工筛选
- 按设备筛选
- 按打卡类型筛选

## 接口职责划分

### 管理端接口

管理端接口通过 oRPC 提供。

建议包括：

- `dashboard.summary`
- `employee.list`
- `employee.create`
- `employee.update`
- `device.list`
- `device.create`
- `device.update`
- `attendanceConfig.get`
- `attendanceConfig.save`
- `faceProfile.list`
- `faceProfile.enqueue`
- `faceProfile.cancel`
- `attendanceRecord.list`

### 设备端接口

设备端接口通过普通 HTTP JSON API 提供。

建议包括：

- `POST /api/device/sync`
- `POST /api/device/enrollment/report`
- `POST /api/device/attendance/upload`

补充约束：

- `POST /api/device/sync` 至少返回设备状态、考勤配置、员工信息和待处理录脸任务
- 设备端接口需要返回稳定的拒绝原因，供设备本地展示或记录

## MVP 边界

当前系统明确不做以下内容：

- 多套考勤方案
- 周几配置
- 轮班
- 请假和补签
- 多设备协同和设备间一致性处理
- 人脸图片存储
- 人脸特征向量存储
- 原始考勤事件审计表
- 复杂权限角色系统
- `apiKey` 重置或轮换
- 员工删除流程

## 结论

当前业务流程的核心思想是：

- 后台负责配置和管理
- 设备负责识别和上传
- 服务端负责最终校验和持久化

整个 MVP 以“能跑通完整考勤闭环”为目标，优先保证实现简单、展示清晰、适合毕设答辩。

补充说明：

- 当前项目部署目标固定为单组织单服务
- 数据库固定使用 SQLite
