# 设备端本地数据设计

## 文档目标

本文档定义 ESP32 设备在当前毕设 MVP 中需要本地保存的数据内容、生命周期和处理规则。

本文档只定义逻辑数据结构和行为，不限定具体存储实现方式。

例如：

- 可以使用文件
- 可以使用键值存储
- 可以使用 Flash 分区
- 可以使用 SQLite 或其他嵌入式存储

无论最终采用哪种实现，设备本地都必须满足本文档描述的数据语义。

---

## 适用前提

本文档基于以下 MVP 前提：

- 单设备演示场景
- 默认使用 `Asia/Shanghai`
- 设备通过 `/api/device/sync` 获取全量配置、员工信息和录脸任务
- 设备通过 `/api/device/enrollment/report` 上报录脸结果
- 设备通过 `/api/device/attendance/upload` 批量上传本地缓存的考勤记录
- 设备需要在断网场景下继续缓存有效打卡结果

---

## 设计原则

### 1. 逻辑先行

- 先明确要保存什么，再决定具体存储介质
- 代码实现时不得因为存储方式不同而改变业务语义

### 2. 全量覆盖优先

- 当前 MVP 的同步策略以全量覆盖为主
- 设备每次同步成功后，可直接覆盖本地配置、员工信息和当前录脸任务列表

### 3. 断网可继续工作

- 断网时设备仍可根据本地配置和本地员工信息继续识别
- 断网时设备仍可缓存待上传考勤记录

### 4. 本地数据规模可控

- 本地拒绝原因和失败日志不应无限增长
- 当前 MVP 推荐保留最近 200 条失败日志

---

## 设备持久化分层架构

当前设备端持久化固定采用 3 层存储，加 1 层运行时内存：

1. `NVS / Preferences`
2. `LittleFS`
3. `SD Card`
4. `PSRAM`（运行时内存，不属于持久化介质）

### 1. NVS / Preferences

用途：

- 保存极少量、最关键、不能依赖文件系统挂载的数据

当前默认保存：

- `deviceCode`
- `apiKey`
- 少量设备级开关
- 存储架构版本号

约束：

- 不保存大对象
- 不保存模板库
- 不依赖 `LittleFS` 或 `SD Card`

### 2. LittleFS

用途：

- 保存设备最小运行态和板载缓存

当前默认保存：

- 考勤配置快照
- 员工信息摘要快照
- 当前录脸任务列表
- 待上传考勤队列
- 接口失败日志
- 模板库版本摘要
- `SD Card` 健康状态摘要

约束：

- 不承担人脸模板库主存储
- 挂载失败时不得自动格式化
- 挂载失败时设备允许降级启动

### 3. SD Card

用途：

- 保存容量型、可增长、可替换的大数据

当前默认保存：

- 员工人脸模板库
- 后续可选导出文件
- 后续可选调试样本

约束：

- 不进入设备核心启动链路
- 缺卡或坏卡时设备继续启动
- 缺卡或坏卡时依赖模板库的识别能力禁用

### 4. PSRAM

用途：

- 作为模板库的运行时加载内存

当前默认行为：

- 启动后把人脸模板从 `SD Card` 加载到 `PSRAM`
- 识别时直接在内存中完成 `1:N` 比对

说明：

- `PSRAM` 不属于持久化介质
- 掉电后 `PSRAM` 中的数据不会保留

## 本地数据总览

当前设备本地需要保存 6 类核心业务数据，外加 2 类存储层辅助状态数据和 1 类容量型模板库数据：

1. 设备身份信息
2. 全局考勤配置
3. 员工信息快照
4. 当前录脸任务列表
5. 待上传考勤队列
6. 接口失败日志
7. 人脸模板库版本摘要
8. `SD Card` 健康状态摘要
9. 员工人脸模板库

按存储层归属如下：

| 数据类型               | 主存储层            | 说明                               |
| ---------------------- | ------------------- | ---------------------------------- |
| 设备身份信息           | `NVS / Preferences` | 设备最核心身份数据，不依赖文件系统 |
| 全局考勤配置           | `LittleFS`          | 同步成功后整体覆盖                 |
| 员工信息快照           | `LittleFS`          | 仅保存识别所需员工基础信息摘要     |
| 当前录脸任务列表       | `LittleFS`          | 当前设备待处理任务状态列表         |
| 待上传考勤队列         | `LittleFS`          | 设备最小运行态的一部分             |
| 接口失败日志           | `LittleFS`          | 设备本地调试和状态展示             |
| 人脸模板库版本摘要     | `LittleFS`          | 保存模板库版本、更新时间或加载摘要 |
| `SD Card` 健康状态摘要 | `LittleFS`          | 保存最近一次挂载/读取状态的摘要    |
| 员工人脸模板库         | `SD Card`           | 人脸模板主存储，不进入服务端数据库 |

## 1. 设备身份信息

### 作用

保存设备访问服务端所需的固定身份信息。

### 逻辑结构

```json
{
  "deviceCode": "DEV-001",
  "apiKey": "plain-api-key"
}
```

### 字段说明

| 字段         | 说明                     |
| ------------ | ------------------------ |
| `deviceCode` | 设备码                   |
| `apiKey`     | 创建设备后获得的明文密钥 |

### 更新时机

- 设备首次初始化时写入
- 当前 MVP 不支持后台任意重置 `apiKey`
- 设备在 bootstrap 激活成功后会把正式 `apiKey` 落到本地

### 删除时机

- 当前 MVP 不主动删除

### 说明

- 这是设备工作所依赖的最核心数据
- 设备重启后必须能重新读出这组信息

---

## 2. 全局考勤配置

### 作用

设备依据本地保存的考勤配置判断识别结果属于 `clock_in` 还是 `clock_out`。

### 逻辑结构

```json
{
  "attendanceConfig": {
    "id": "default",
    "workStartMinute": 540,
    "workEndMinute": 600,
    "offStartMinute": 1080,
    "offEndMinute": 1140,
    "updatedAt": 1743158400000
  },
  "syncedAt": 1743158400000
}
```

### 更新时机

- 每次 `/api/device/sync` 成功后整体覆盖

### 删除时机

- 若同步成功且服务端返回 `attendanceConfig = null`，则清空本地配置

### 断网行为

- 断网时继续使用最近一次成功同步的配置
- 若本地没有配置，则设备不得生成待上传考勤记录

### 说明

- 时间段解释固定使用 `Asia/Shanghai`
- 时间段判定规则固定为 `[start, end)`

---

## 3. 员工信息快照

### 作用

保存设备本地识别所需的员工基础信息。

### 逻辑结构

```json
{
  "items": [
    {
      "id": "emp_001",
      "code": "20230001",
      "name": "张三",
      "updatedAt": 1743158400000
    }
  ],
  "syncedAt": 1743158400000
}
```

### 更新时机

- 每次 `/api/device/sync` 成功后整体覆盖

### 删除时机

- 仅在同步成功且服务端返回空员工列表时整体清空

### 断网行为

- 断网时继续使用最近一次成功同步的员工快照

### 说明

- 当前 MVP 使用全量员工同步，不做增量合并
- 同步成功后，本地旧员工快照直接被新快照替换

---

## 4. 当前录脸任务列表

### 作用

保存当前设备待处理的录脸任务列表。

### 逻辑结构

```json
{
  "items": [
    {
      "taskId": "fp_001",
      "employeeId": "emp_001",
      "employeeCode": "20230001",
      "employeeName": "张三",
      "status": "pending",
      "createdAt": 1743158400000,
      "updatedAt": 1743158400000
    }
  ]
}
```

### 更新时机

- 每次 `/api/device/sync` 成功后，用服务端返回的 `enrollmentTasks` 全量覆盖本地值
- 单条录脸上报成功后，从本地列表移除对应任务
- 单条录脸上报返回 `TASK_CANCELLED` 后，从本地列表移除对应任务

### 删除时机

- 服务端同步返回空数组时清空
- 当前最后一条任务处理完成后清空

### 断网行为

- 断网时可继续保留最近一次同步得到的待处理任务列表
- 若设备在断网状态下完成录脸，应等待联网后再上报

### 说明

- 当前设计允许一台设备同一时刻持有多条 `pending` 录脸任务
- 本地只需要保存当前待处理任务列表，不需要保存历史录脸任务列表

---

## 5. 待上传考勤队列

### 作用

缓存设备在断网或服务端暂不可用时未完成上传的有效考勤记录。

### 逻辑结构

```json
{
  "items": [
    {
      "clientRecordId": "local-001",
      "employeeId": "emp_001",
      "recognizedAt": 1743158400000,
      "type": "clock_in",
      "localDate": "2026-03-28",
      "createdAt": 1743158401000,
      "lastAttemptAt": null,
      "lastResultCode": null
    }
  ]
}
```

### 字段说明

| 字段             | 说明                                              |
| ---------------- | ------------------------------------------------- |
| `clientRecordId` | 本地唯一记录 ID，用于匹配上传响应                 |
| `employeeId`     | 员工 ID                                           |
| `recognizedAt`   | 识别时间                                          |
| `type`           | `clock_in` 或 `clock_out`                         |
| `localDate`      | 由 `recognizedAt` 按 `Asia/Shanghai` 计算出的日期 |
| `createdAt`      | 本地入队时间                                      |
| `lastAttemptAt`  | 最近一次尝试上传时间                              |
| `lastResultCode` | 最近一次服务端返回的结果码                        |

### 入队时机

- 本地识别成功
- 当前时间命中有效打卡时间段
- 根据本地去重规则判定应保留该记录

### 本地去重规则

队列中按以下组合键去重：

- `localDate`
- `employeeId`
- `type`

处理方式：

- 若组合键不存在，则直接入队
- 若组合键已存在且新记录更早，则替换为更早的 `recognizedAt`
- 若组合键已存在且新记录更晚或相同，则丢弃新记录

### 上传时机

- 设备检测到网络可用时
- 设备主动触发批量上传时

### 上传后处理规则

当 `/api/device/attendance/upload` 顶层请求成功时：

- `saved`：从本地队列删除
- `updated_earlier`：从本地队列删除
- `ignored_duplicate`：从本地队列删除
- `rejected`：记录失败日志后，从本地队列删除

当 `/api/device/attendance/upload` 顶层请求失败时：

- 所有记录保留在本地队列
- 为每条已提交记录更新 `lastAttemptAt`
- 将顶层失败码写入 `lastResultCode`

### 断网行为

- 断网期间队列持续累积
- 恢复联网后按本地队列顺序上传

### 说明

- 当前 MVP 建议上传成功后立即删除对应本地记录
- 当前 MVP 不要求保留已成功上传的本地考勤历史

## 6. 接口失败日志

### 作用

记录设备调用服务端接口时的关键失败信息，供设备本地展示或调试。

### 逻辑结构

```json
{
  "items": [
    {
      "id": "log-001",
      "api": "/api/device/attendance/upload",
      "occurredAt": 1743158400000,
      "code": "ATTENDANCE_NOT_IN_WINDOW",
      "message": "recognizedAt is not in valid attendance window",
      "relatedId": "local-001"
    }
  ]
}
```

### 字段说明

| 字段         | 说明                                               |
| ------------ | -------------------------------------------------- |
| `id`         | 本地日志 ID                                        |
| `api`        | 发生错误的接口路径                                 |
| `occurredAt` | 发生时间                                           |
| `code`       | 服务端返回的稳定错误码                             |
| `message`    | 错误说明                                           |
| `relatedId`  | 可选关联对象 ID，例如 `taskId` 或 `clientRecordId` |

### 写入时机

- 任一设备接口返回顶层失败
- 录脸上报返回 `TASK_CANCELLED`
- 考勤上传单条记录返回 `rejected`

### 清理规则

- 当前 MVP 推荐只保留最近 200 条
- 超出上限时删除最早日志

### 说明

- 本地失败日志用于辅助调试
- 它不是服务端审计数据

---

## 7. 人脸模板库存储原则

### 作用

保存设备本地 `1:N` 识别所需的员工人脸模板库。

### 当前默认规则

- 模板主存储固定为 `SD Card`
- 模板只保存在设备本地，不进入服务端数据库
- 当前方案只保存模板，不保存人脸原始图片
- 每位员工当前只保留 1 份模板
- 当前规模假设为 `50` 人以内

### 与其他本地数据的关系

- 员工基础信息快照与模板库是两类不同数据
- 员工基础信息快照保存在 `LittleFS`
- 人脸模板库保存在 `SD Card`
- 设备识别时需要同时依赖员工信息摘要和本地模板库

### 运行时规则

- 启动后可把模板库全量加载到 `PSRAM`
- 识别时默认在 `PSRAM` 中完成 `1:N` 比对
- 当前文档不固定模板二进制格式
- 当前文档不固定具体识别模型

### 当前不做

- 保存人脸原始图片
- 服务端模板同步
- 多模板/多视角模板管理
- 模板历史版本管理

---

## 8. 模板库版本与存储状态摘要

### 作用

保存模板库加载和外部存储状态所需的辅助信息，供设备启动诊断、状态展示和后续实现使用。

### 当前默认内容

- 模板库版本号或更新时间摘要
- 最近一次模板库加载结果
- 最近一次 `SD Card` 挂载或读取状态摘要

### 说明

- 这些摘要数据属于设备最小运行态，因此保存在 `LittleFS`
- 它们不是模板库本体

---

## 启动与降级规则

设备启动时必须分别检测 `NVS`、`LittleFS` 和 `SD Card`，不得再用单一布尔值表达“存储是否正常”。

### 1. NVS / Preferences 异常

- 设备身份不可用
- 设备不能进入正常工作态
- 设备不能继续依赖设备身份的同步、录脸上报和考勤上传流程

### 2. LittleFS 异常

- 设备允许启动和联网
- 设备本地状态持久化能力降级
- 不得自动格式化文件系统
- 不得静默清空本地快照、待上传队列或失败日志

### 3. SD Card 异常

- 设备允许启动
- 设备最小运行闭环继续由 `NVS` 和 `LittleFS` 维持
- 人脸模板库不可用
- 依赖模板库的本地识别能力禁用

### 4. 运行级别划分

当前推荐把设备运行态理解为以下 3 个层级：

- `Core Ready`：`NVS` 正常，设备身份可用
- `Runtime Ready`：`LittleFS` 也正常，设备具备完整本地状态能力
- `Recognition Ready`：`SD Card` 也正常，设备具备模板库和识别能力

---

## 启动与运行时数据流

### 1. 启动时

1. 读取本地设备身份信息
2. 检测 `LittleFS` 是否可用
3. 若 `LittleFS` 可用，则读取本地考勤配置
4. 若 `LittleFS` 可用，则读取本地员工信息快照
5. 若 `LittleFS` 可用，则读取本地当前录脸任务列表
6. 若 `LittleFS` 可用，则读取本地待上传考勤队列
7. 检测 `SD Card` 是否可用
8. 若 `SD Card` 可用，则加载人脸模板库
9. 尝试调用 `/api/device/sync`

### 2. 同步成功后

1. 覆盖本地考勤配置
2. 覆盖本地员工信息快照
3. 覆盖本地当前录脸任务列表
4. 保留待上传考勤队列不变

### 3. 识别成功后

1. 根据本地考勤配置判断打卡类型
2. 若命中有效时间段，则写入待上传考勤队列
3. 若不命中有效时间段，则不入队

### 4. 上传成功后

1. 根据逐条返回结果处理本地队列
2. 需要记录的失败原因写入本地失败日志

---

## 当前不做

- 本地保存历史录脸结果
- 本地保存成功上传后的考勤历史
- 员工信息增量同步
- 考勤配置版本比对
- 失败日志上传回服务端
- 设备本地审计报表
- 人脸原始图片保存
- 服务端模板同步
- 多模板/多视角模板管理
